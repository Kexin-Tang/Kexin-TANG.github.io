NoSQL 数据库

NoSQL (No only SQL)是一个高性能的 key-value 非关系型数据库。

Redis 优势

Redis 有众多优势：

  1. 性能好：可以用于减小CPU压力和数据库的读写压力
  2. 数据类型丰富
  3. 原子性：即单个操作是原子性的，要么成功要么完全不执行；多个操作可以使用事务完成原子性操作

    注：Redis 支持事务，但是和 MySQL 的事务不同，并不支持ACID特性。

  4. 持久化数据库：运行在内存，但是可以将数据持久化到磁盘

Redis 主要适用于“高并发、大数据、可扩展”的情形，但是不适用于“需要大量事务操作和结构化查询”的任务中。

redis VS memcache

Redis 和常用的 memcache 有很大的不同：

  1. memcache 不能持久化，只能将数据保存在内存中，断电后数据消失
  2. memcache 只能支持String类型
  3. memcache 使用 “串行+多线程+锁” 的方式实现并发，而 redis 使用 “单线程+多路IO复用” 实现并发

为什么 redis 支持高并发并效率高

  1. redis 基于内存，免去了磁盘读写，减小了DB的IO压力
  2. redis 单线程，不受CPU的限制，也省去了上下文切换
  3. redis 多路复用，可以处理并发请求 (使用了epoll)

Redis 数据类型和基本操作

数据库设置

Redis 默认有16个库

  • select <dbid>来进行数据库切换
  • keys *查看所有的keys
  • exists <key>查看某个key是否存在
  • del <key>删除某个key
  • unlink <key>异步删除某个key
  • type <key>查看key类型
  • expire <key> <time>设置某个key的生命周期
  • ttl <key>查看当前剩余时间，-1为永不过期，-2为已过期

String

是二进制安全的一种数据类型，可以存储任何“可转换为string类型”的数据，其底层是SDS(simple dynamic string，简单动态字符串)。该数据结构会预先分配一定的capacity，然后在空间不足时分配更多capacity。

capacity最大为512MB，小于1MB时空间进行翻倍，大于1MB时每次多分配1MB

  • set <key> <value>设置键值对
  • setnx <key> <value>在key不存在时才设置
  • get <key>获取值
  • append <key> <str>将str附加到值的尾部
  • strlen <key>获取值的长度
  • incr/decr <key>值++/–
  • incrby/decrby <key> <step>值+step/-step
  • mset <k1> <v1> <k2> <v2> ...设置多个键值对
  • mget <k1> <k2>获取多个值
  • getrange <key> <start> <end>取key[start, end]，注意两边界可取
  • setrange <key> <start> <value>在start处插入一个value
  • getset <key> <value>获取旧值，设置新值

List

单键多值，底层为双向链表(quickList)，在数据量少时，使用一块连续内存，称为zipList，在数据量多时，将多个zipList作为节点，构建双向链表，称为quickList。

  • lpush/rpush <key> <v1, v2, ...>从左侧/右侧压入数据
  • rpoplpush <k1> <k2>从k1的右侧弹出元素放入k2的左侧
  • lrange <key> <start> <end>key[start, end]
  • lindex <key> <index>key[index]
  • llen获取长度
  • linsert <key> before/after <value> <newvalue>在key[value]前/后插入一个newvalue
  • lrem <key> <n> <value>

Set

与list相似，但是不允许重复，底层是字典(hash表)，增删查改都是O(1)。

  • sadd <key> <v1, v2, ...>
  • smembers <key>
  • sismember <key> <value>if value in key?
  • scard <key>返回个数
  • srem <key> <v1, v2, ...>删除
  • spop <key>随机弹出一个
  • srandmember <key> <n>随机抽取n个
  • smove <k1> <k2>从k1移动一个到k2
  • sdiff/sinter/sunion <destkey> <k1> <k2>将k1与k2进行操作放入destkey

Hash

存储的是 {key: {field: value, field: value}} 的结构。

  • hset <key> <field> <value>
  • hget <key> <field>
  • hmset <key> <field1> <value1> <field2> <value2>...
  • hexists <key> <field>
  • hvals/hkeys <key>查看所有value/key

Zset

与set类似，但是是有序的，因为其为每个成员设置了一个score用于排序。底层使用了两种数据结构：

  1. hash: {key: {value: score}}
  2. 跳表
  • zadd <key> <s0> <value0> <s1> <value1> ...
  • zrange <key> <start> <end> [withscores]
  • zrangebyscore <key> <min> <max> [withscores]从大到小排序
  • zrevrangebyscore ...反序
  • zcount <key> <min> <max>范围内的元素个数
  • zrank <key> <value>返回排名，从0开始计数
  • zrem <key> <value>删除

Bitmaps

实际上是字符串，但可以视为只能存储0和1，其下标被称为“偏移量”，用于提高内存使用率。

  • setbit <key> <offset> <value>
  • getbit <key> <offset>
  • bitop and/or/not/xor <destkey> <key...>
  • bitcount <key> [start] [end]用于记录为1的个数，注意此处是针对8个bits，即一个byte为单位

HyperLogLog

用于做基数统计。在输入数据巨大时，计算基数所需的空间总是固定且较小的，比如在Redis中可以用12KB计算264个基数。

基数：不重复的数字个数

  • pfadd <key> <element ...>
  • pfcount <key>
  • pfmerge <destkey> <key1, key2 ...>将多个Log进行合并

GEO

主要用于保存地理信息，可以计算直线距离等

事务&锁

Redis 事务是一个单独的隔离操作，所有命令序列化、按序执行，不会被插手打断。之所以要引入事务，就是为了将多个小操作合并成一个大操作，且“一次性”执行完整个大操作。

基本步骤

  1. multi – 用于组合小操作，此时只是组合，并未执行。如果出现错误，则所有的操作均被抛弃，无法运行任何一个。
  2. exec – 执行队列。如果有错误，则只会执行正确的步骤，并不会像MySQL一样回滚。
  3. discard – 放弃之前的操作。

解决冲突

由于现实生活中都是并发执行的，所以可能出现多个对同一资源的诉求，这被称作冲突。解决冲突的方法就是加锁。

乐观锁

乐观锁认为资源不会轻易受污染，所以放任多个线程同时访问，如果访问结束后出现冲突再解决。

具体方法是：使用版本号，对资源操作后更新版本号，其他已经占用资源的线程如果想更改资源，发现版本号不同后就无法操作。

悲观锁

悲观锁认为资源很容易就被污染，所以每一步操作都要加以限制。

具体方法就是：一旦某个进程控制了资源，就给他加上锁，别的进程就只能等待。当使用完资源后，进程再解锁，别的进程再开始竞争。

其逻辑简单，但是相当于是串行实现多线程，效率低下。

代码实现

使用multi开启组队模式，组队完成后可以使用exec进行执行或discard放弃组队。而 Redis 中是不支持直接使用悲观锁的，所以加锁都是乐观锁。具体操作就是使用watch <key>完成对某个数据的监视，在事务执行前，如果被监视的数据发生了变化，则事务不能执行。

三大特性

  1. 单独隔离操作
  2. 无隔离级别
  3. 不保证原子性 – 只能保证不被打断，一旦出错是不会rollback的

Pipeline

Redis客户端执行一条命令分为以下四个步骤:

  1. 发送命令
  2. 命令排队
  3. 命令执行
  4. 返回结果

其中,第一步+第四步称为Round Trip Time(RTT,往返时间)。

如果想要执行多个步骤，则需要耗费多个RTT，如果能把这些步骤在一个RTT内一起发送，就会非常有效率。这样的实现机制就是Pipeline。

Pipeline和事务的差别

  1. Pipeline是将命令打包好一起发送，服务器接受后再一条一条的执行，处理完再将这些命令的结果一并返回；但是事务是在组队后发送多个操作，服务器在组队完毕后一口气执行完，最后再返回结果。(pipeline是将命令打包一次发送，服务器多次执行；事务是命令分开发送，服务器一并执行)
  2. pipeline只不过是将命令一起发送，对服务器而言是不知道客户端是否使用了pipeline；而事务是服务器支持的，客户端的命令发送仍然是分离的，但是服务器会将组队的一起执行。(pipeline是客户端行为，事务是服务端行为)
  3. 执行事务时会阻塞其他任务，而pipeline不会
  4. 事务是原子性的，pipeline不是

Lua脚本

Lua是一种非常轻量级的脚本语言，一般使用C语言编写，其设计目的是为了嵌入应用程序中，从而为应用程序提供灵活的扩展和定制功能。

为什么要使用Lua

  1. 减少网络开销(类似pipeline)：多个请求通过脚本一次发送，减少网络延迟
  2. 原子操作(类似事务)：将脚本作为一个整体执行，中间不会插入其他命令，无需使用事务
  3. 复用：客户端发送的脚本永久存在redis中，其他客户端可以复用脚本
  4. 可嵌入性：可嵌入JAVA，C#等多种编程语言，支持不同操作系统跨平台交互

如何使用Lua

  1. EVAL script numkeys key [key ...] arg [arg ...]

其中script是脚本的内容或路径，numkeys为传入的key的个数，然后key和arg分别对应Lua脚本中的 KEYS[1] KEYS[2] … 和 ARGV[1] ARGV[2] …

  1. SCRIPT LOAD script

返回脚本的SHA1校验和，不执行

  1. EVALSHA sha1 numkeys key [key ...] arg [arg ...]

根据传入的 sha1 校验和执行脚本

持久化

RDB

RDB (Redis Database)是在指定时间间隔内将数据快照(snapshot)从内存中转移到磁盘中，恢复时将快照转至内存中。

Redis中RDB的配置

默认在开启服务器的路径下生成一个dump.rdb文件，用于保存快照，其中可以设置时间间隔和达到的目标，如3600秒内更新了100个数据才会进行持久化。

如果3600s内更新了101个文件，那么前100个会进行持久化，多余的1个会算入下一个3600s内。

还有一些设置是否自动持久化、磁盘无法写入是否终止等操作的，可以自己摸索。

如何持久化

Redis会单独创建一个子进程用于持久化，会先将数据写入一个临时文件，当持久化结束后再用临时文件替换上一次持久化的文件。整个过程中，主进程不进行任何IO操作。

由于需要将临时文件全部写入后才会更换原来的持久化文件，如果当写临时文件时出现宕机等，导致临时文件没有写完，那么持久化文件仍保存了上一次的数据，新的数据就丢失了。

优缺点

优点：

  • 适合大规模数据，因为只需要父进程fork一个子进程，然后就不用管子进程
  • 节省磁盘空间
  • 恢复速度快

缺点：

  • fork需要生成临时文件，会占用额外空间
  • 最后一次持久化后的数据可能丢失
  • 虽然使用写时拷贝技术，但是如果数据庞大时还是比较消耗性能

AOF

AOF (Append Only File)会以日志形式记录每个写操作(增量保存，不会记录读操作)，只许追加文件但不可以改写文件。Redis启动时会将日志内容重新按序执行一次以恢复/加载数据。

默认情况下AOF是不开启的，如果RDB和AOF同时开启，Redis会选择使用不会丢失信息的AOF。

AOF同步频率

  • appendfsync always：始终同步，每次写入操作都会记录到日志中，性能差但是完整性好
  • appendfsync everysec：每秒同步一次
  • appendfsync no：不主动同步，同步的时机由OS控制

Rewrite数据压缩

将多个步骤压缩成较少的步骤，如set k1 v1和set k2 v2合并成mset k1 v1 k2 v2进行保存。

当AOF文件持续增长而过大时，会fork一个新的进程来重写文件(先创建临时文件再覆盖)，实际上就是把rdb的快照以二进制形式加在新的aof的头部，作为历史数据汇总，替换流水账。

流程

  1. 客户端写命令被追加到AOF缓冲区
  2. AOF缓冲区根据同步的频率将内存→磁盘
  3. AOF文件超过大小时进行rewrite
  4. Redis下次启动时会加载AOF的写操作完成数据加载

优缺点

优点：

  • 备份机制更加稳健，丢失数据概率低
  • 日志文件易读，且有日志文件纠错

缺点：

  • 占用磁盘空间
  • 恢复和加载数据慢
  • 同步频率设置过高会造成巨大压力

选择RDB还是AOF？

  • 建议两者都开启，搭配使用
  • 如果不追求数据完整性，建议使用RDB
  • 不建议单独使用AOF，因为可能存在日志BUG无法恢复数据
  • 如果只是做内存缓存，则可以都不使用

主从复制

主机数据更新后根据配置和策略，自动同步到备机的master/slave机制，Master以写为主，Slave以读为主

为什么要使用主从复制

  1. 读写分离，提高效率
  2. 容灾快速恢复：由于一般有多个从服务器(即可以从多个地方读)，所以一旦一个服务器宕机，还可以依赖别的服务器
